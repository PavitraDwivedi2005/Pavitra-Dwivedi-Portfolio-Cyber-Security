Building a Privacy-Preserving Scam Detector: What I Learned Combining AI and Security

By Pavitra Dwivedi May 4, 2026 12 min read

A few months ago I started working on what has become my most challenging and also my most rewarding project. The idea is simple to explain but hard to build: a system that can detect voice-based scams happening in real time using AI but without ever storing the caller's voice data. It sounds like it should not work. How do you analyse something you do not keep? That contradiction is exactly what makes this problem interesting to me.

This is not a tutorial. I am still figuring things out. This is more of a project diary from a student who ran into a lot of problems and learned from most of them.

Why This Problem Matters

"Digital arrest" scams are becoming really common in India. Scammers call people pretending to be government officials. They threaten arrest and pressure victims into transferring money. The calls are designed to create panic and they work surprisingly well.

The obvious solution would be to just record all calls and analyse them later. But that creates a huge privacy problem. You cannot protect people by watching everything they do. So the question I kept coming back to was: can you detect bad patterns in real time using only temporary data that gets thrown away right after?

How the System Works

The basic idea behind the architecture is simple: process in memory extract what you need then delete the audio. Here is how it works step by step:

  • Audio capture — the system grabs small chunks of audio from the call. Just a few seconds at a time.
  • Speech-to-text — each chunk gets transcribed locally using Vosk which is an offline speech recognition model. So no audio ever leaves the device.
  • Feature extraction — the transcript gets checked for certain patterns. Things like urgency language impersonation phrases and known scam tactics.
  • Risk scoring — a machine learning model looks at all the features and gives a risk score. The score is explainable meaning you can see exactly why it flagged something.
  • Alert — if the score is too high the user gets a warning.

The important part is that the raw audio never gets saved to disk. The transcript exists only in memory and gets deleted right after the features are extracted. The only thing that stays is the anonymised feature data and the risk score.

Problems I Did Not Expect

Audio Capture on Windows is Painful

This was my first real lesson in how different things are when you work with actual hardware. Capturing system audio on Windows means using something called WASAPI loopback. The documentation for it is not great and it behaves differently depending on what audio drivers you have installed.

I spent days trying to figure out why my audio stream was recording silence. Turns out the Nahimic drivers on my laptop were intercepting the audio output before my code could grab it. The fix was building a capture module that checks what audio setup the user has and adjusts itself. Not pretty but it works.

Real-Time Processing Without Breaking the Call

The system has to analyse audio while the call is still happening. And it cannot slow down the call or make it sound choppy. That means everything has to be non-blocking and fast enough to keep up.

I used Python's multithreading for this. Audio capture transcription and analysis all run at the same time in different threads. I will be honest I underestimated how complicated this would be. Thread safety shared buffers and making sure everything shuts down properly taught me more about concurrent programming than any class ever did.

Making the AI Explain Itself

A model that just says "this call is suspicious" is not helpful. People need to know why. So the risk scoring system breaks down its decision into parts: urgency language detected here impersonation pattern found there pressure tactic used at this point financial request trigger at that point.

This requirement changed the whole ML architecture. Instead of one big model that gives a single answer I use separate scoring for each type of scam indicator. Then they get combined with weighted aggregation. It is more work but the results actually make sense to a person reading them.

What I Learned About Privacy Engineering

Before this project privacy was just a word I would see in articles. Now it is something I think about in very concrete terms. Here are the principles that guide how I build this system.

Treat All Audio Data as Sensitive

Nothing gets written to disk unless it has been through an anonymisation process. This is influenced by what I learned at my internship at IQSec Labs where I built data anonymisation pipelines for a living.

Keep Data in Memory for Seconds Not Minutes

Audio chunks exist in memory for just a few seconds. The moment features are extracted the raw data gets overwritten. There is no temporary file sitting around that someone could recover later.

Everything Happens on the Device

All processing happens locally. No audio no transcripts no feature data gets sent to any server anywhere. The Vosk model runs on the user's machine. The ML model runs on the user's machine. Everything stays local.

The Next Step: Federated Learning

The next thing I want to add is federated learning. The basic idea is that the ML model can get better by learning from patterns across many users but no user's data ever leaves their device. Instead only the model updates get shared and combined. Getting this right is what I am working on now and it is hard but really exciting.

The Tech Stack

For anyone curious about what I am using:

  • Python — the main language for everything
  • Vosk — offline speech-to-text that supports English and Hindi
  • PyAudio and sounddevice — for audio capture through WASAPI loopback
  • scikit-learn — for the risk scoring ML model
  • Threading — for running audio capture transcription and analysis all at the same time
  • EMA smoothing — for making the real-time risk scores less jumpy

What This Project Has Taught Me

The biggest thing I have learned is not technical. It is about thinking in systems. When you are building something that touches audio processing ML privacy and real-time computing all at once you cannot think about one part at a time. Every decision you make in one module affects everything else.

The second thing: constraints make you more creative. The privacy constraint of not storing voice data forced me to rethink the whole architecture. It would have been way easier to just save everything and look at it later. But the constraint pushed me toward a better design.

The third thing: just ship it even if it is rough. The first version of this system was bad. The audio capture did not work half the time the risk scoring was basic and there was no UI. But getting it working end-to-end even poorly gave me a foundation to improve. I am now on a version that is good enough for a copyright filing.

Frequently Asked Questions

How does the system detect scams without storing voice data? +

It processes audio in small chunks that only exist in memory. Each chunk gets transcribed locally then the system extracts features like urgency markers and impersonation patterns. After that the raw audio and transcript get deleted. The only things that stay are anonymised feature data and the risk score. The actual voice never touches the hard drive.

What is federated learning and why does it matter here? +

Federated learning is a way of training a machine learning model across many devices without putting all the data in one place. Each device learns from its own local data and only shares model weight updates not the actual data. This means the model can improve from seeing more patterns while keeping all user data completely private on their own device.

Can this work for languages other than English? +

Yes. Vosk has offline models for many languages including Hindi which is important for India where these scams are common. The feature extraction part needs to be adjusted for each language's scam patterns but the core system does not care what language is being spoken. Hindi support is something I am actively working on.

What skills do I need to build something like this? +

You will need solid Python skills an understanding of how audio processing works some familiarity with ML concepts like feature engineering and classification and experience with multithreaded programming. Understanding privacy engineering principles like data minimisation and transient processing also helps a lot. My advice would be to start small. Build a basic speech-to-text pipeline first then add analysis layers one at a time.
